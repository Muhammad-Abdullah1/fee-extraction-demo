# Construction / Development Fee Extraction — Proof of Concept

A small demo that proves the core idea: pull real fees from official
city/county sources, store them in a structured form, and keep every
extracted number tied to the exact line of source text it came from.

The bar is **traceability**, not coverage. Every number you see in the
UI is one click away from the literal line in the original PDF.

---

## What it proves

1. **Real extraction works** — **2,217 fee records** pulled from real,
   public PDFs and ordinances published by the City of Calimesa,
   Riverside County, the Riverside County Department of Environmental
   Health, and the Eastern Municipal Water District.
2. **Structured storage works** — `sources`, `fees`, and `change_logs`
   tables; one row per fee, every fee linked to a source row.
3. **Source traceability works** — every `fees` row stores the exact
   `raw_snippet` it was parsed from, plus a link back to the source's
   public URL. The fee detail page shows the snippet in a `<pre>` block
   right next to the parsed amount.
4. **No invented numbers** — the parsers only emit a fee record when
   they have a literal `$N` token (or `N%`) in the source. AI is **not**
   used anywhere in the value path.
5. **Honest uncertainty** — same-name-different-amount rows produce
   `conflicting` flags rather than silent picks. OCR-derived numbers
   are flagged `needs_review` until a human verifies them. Image-only
   PDFs and missing sources are kept as `pointer` source rows with a
   note explaining the blocker.

## Setup

Requires Python 3.10+. The Calimesa DIF source needs OCR; install
Tesseract and Ghostscript with Homebrew:

```bash
brew install tesseract ghostscript
pip install ocrmypdf

cd demo
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# (only needed once) OCR the image-only Calimesa DIF PDF
ocrmypdf --skip-text \
  fixtures/calimesa_dif_schedule_2017.pdf \
  fixtures/calimesa_dif_schedule_2017_ocr.pdf

# build the SQLite DB and ingest all configured sources
python -m scripts.run_extraction --reset

# run the web UI
uvicorn app.main:app --reload
# then open http://127.0.0.1:8000
```

## Deploy on Railway

This repo includes a Railway-ready startup path:

- `railway.json` tells Railway to start `./scripts/railway-start.sh`
- `scripts/railway-start.sh` sets `FEES_DB_PATH`, creates the DB directory,
  bootstraps the SQLite database from the bundled `fixtures/` on first deploy,
  and then starts `uvicorn`

Recommended Railway setup:

1. Create a new Railway project from this GitHub repo.
2. Add a `Volume` and mount it at `/app/data`.
3. Set `FEES_DB_PATH=/app/data/fees.db`.
4. Deploy.

On first deploy, the app will build `fees.db` automatically. Later
deploys reuse the persisted volume and skip re-ingestion.

`run_extraction.py` reads from `fixtures/*.pdf` if present (the demo
ships with cached copies of the public PDFs). If a fixture is missing
it will download the file from the official URL on first run.

CLI options:

```bash
python -m scripts.run_extraction --reset                    # wipe DB and reload all
python -m scripts.run_extraction --source rivco_dif_ord_659_14
```

## Database schema

Mirrors the spec.

```text
sources(id, url, title, jurisdiction, department, source_type,
        fetched_at, raw_content, raw_snippet, local_path, notes)

fees(id, source_id → sources.id, fee_name, amount, amount_text,
     unit, department, jurisdiction, category, confidence,
     validation_flag, raw_snippet, notes, extracted_at)

change_logs(id, source_id, fee_id, field_name, old_value,
            new_value, detected_at)
```

`source_type` is one of `pdf`, `html`, or `pointer`. `pointer` rows
hold zero fees but document a relationship or known blocker (e.g.
"Calimesa EH coverage comes from Riverside County DEH").

`amount` is the parsed float (nullable when the source value is a
percentage, "Actual Cost", "No Fee", or "N/A"); `amount_text` is the
literal token from the source (`"$ 223"`, `"$1,310.00"`, `"3.75%"`,
`"No Fee"`).

`change_logs` is wired up in the model layer for future "did the fee
change since last extraction" support; the demo doesn't write to it
yet — it's there so you can see the planned shape.

## Validation flags

| Flag | Meaning |
|---|---|
| `verified` | Parsed cleanly from a labeled fee row with a numeric amount in a text-extractable PDF. |
| `needs_review` | Amount couldn't be parsed numerically, the source has known formatting noise, or the value came from an OCR layer. |
| `conflicting` | Within one source, the same `(department, subheader, fee_name, unit)` produced two or more different amounts. Both rows are kept. |
| `incomplete` | Reserved. Kept in the schema for partial extractions. |

`confidence` is a float in `[0, 1]` reflecting **extraction clarity**,
not model confidence: 0.95 for a clean tabular cell, 0.9 for a
recognised line in the master fee schedule, 0.6 for an OCR-derived
value, lower still when section context or unit is missing.

Current totals (after a clean run): roughly **1,830 verified**,
**260 conflicting**, **130 needs_review**, **0 incomplete**.

## How traceability works (concretely)

For every fee row in `fees`:

- `source_id` → exactly one `sources` row, which carries the public
  `url`, the `title` of the document, when it was `fetched_at`, the
  `local_path` of the cached file, and the full `raw_content`.
- `raw_snippet` on the fee row stores the literal line (or
  `Row: a | b | c` for table cells) the amount was read from.
- `extracted_at` records when the demo's parser produced this row.

The fee detail page (`/fees/{id}`) renders all of the above on one
screen: parsed amount, raw snippet in a `<pre>` block, plus the source
link clickable straight back to the public PDF.

To eyeball traceability:

1. Open `/`, search for "Fire Booster Pump" or filter by category
   "Water".
2. Click any row. The "Raw source evidence" block shows the exact
   line from the PDF.
3. Click the source URL. The browser opens the official PDF — the
   amount in the snippet appears verbatim in the document.

## Per-source extraction notes

### `calimesa_master_fee_2025` (886 fees)
Line-based regex over the text layer. Recognises top-level sections
(Fire, Building, Planning, Public Works, etc.) and uses the most
recent subheader to disambiguate same-name rows under different
sub-systems. Flags `conflicting` whenever the same
`(department, subheader, fee_name, unit)` appears with two amounts in
the same source.

### `rivco_eh_community_events_2024` (55 fees)
Clean, table-based extraction (`pdfplumber.extract_tables()`). Three
tables: Group Permit, Individual Vendor, Event Organizer. The PDF
contains one source typo (`"$5, 120.00"`) which the parser preserves
verbatim and flags `needs_review`.

### `rivco_dif_ord_659_14` (1,089 fees)
Walks all "Maximum Justified Development Impact Fee Schedule, …" tables
in the ordinance and emits one record per `(area_plan, land_use,
component)` cell. 18 area plans × 7 land uses × 11 components.
"Fire Protection" cells route to category `Fire Department`; traffic,
parks, trails, and flood control route to `Land Use Management`.

### `emwd_consolidated_rates_2026` (115 fees)
Walks every page that mentions a rate program and pulls rows
containing exactly three dollar amounts (the 1/1/2025, 1/1/2026, and
1/1/2027 columns). The 1/1/2026 column is recorded as the canonical
fee; all three columns remain in `raw_snippet`.

### `calimesa_dif_2017` (72 fees, OCR-derived, all `needs_review`)
The original PDF is an image scan with no extractable text. We OCR
it with `ocrmypdf` and parse the resulting text layer for the Exhibit A
fee schedule on page 9. **Every value extracted from the OCR layer
is flagged `needs_review`** with a note instructing the reviewer to
verify against the original image. The OCR introduced a few obvious
errors (e.g. `$0,157` for `$0.157`); the parser auto-corrects only the
most unambiguous comma↔decimal swap and records the fix in `notes`,
leaving everything else literal.

### `calimesa_eh_pointer` (0 fees, pointer)
Calimesa does not run its own Environmental Health program — it is
served by Riverside County DEH. This row documents the relationship.
The applicable fees are the ones in `rivco_eh_community_events_2024`.

### `yvwd_rates_blocker` (0 fees, pointer)
Yucaipa Valley Water District provides Calimesa's water and sewer
service. YVWD's current rate ordinance is not exposed at a stable
public URL; rates appear inside meeting packets under
`documents.yvwd.dst.ca.us`. To complete this gap a human needs to
identify the latest rate ordinance and add a parser for it.

## Assumptions

- **SQLite, not Postgres.** The deliverables list says SQLite and the
  storage line says Postgres; we picked SQLite because the demo is
  meant to run with a single command and no infra. Swapping is a
  one-line change in `app/db.py` (`DATABASE_URL`).
- **No AI in the value path.** The spec says AI may be used for
  cleaning and structuring but not for deciding amounts. To keep this
  guarantee bulletproof we did not call any AI model — parsers are
  deterministic regex + table walks. A future iteration could add a
  cleanup pass for messy section headers, but it would never see the
  numeric column.
- **OCR introduces uncertainty.** Anything from the Calimesa DIF
  source is `needs_review` by construction. The user can read the raw
  OCR snippet and visually compare to the original document.
- **Cached fixtures preferred.** The demo ships with the real PDFs in
  `fixtures/` so it's reproducible offline. The fetcher will re-download
  from the official URL if the cache is missing.
- **Playwright stub only.** All current sources are static PDFs, so
  the included fetcher is plain `requests`. A Playwright fallback hook
  is left in `extractor/fetch.py` for future JS-rendered pages.

## Blockers and messy source issues found

These are real, not hypothetical — they came up while building. The
system surfaces each one clearly rather than hiding it.

1. **YVWD rate ordinance — no stable public URL.** Calimesa's water
   provider does not publish a current consolidated rate PDF that is
   reachable by URL. Rates exist inside individual board-meeting
   packets under `documents.yvwd.dst.ca.us`. Recorded as a `pointer`
   source so the gap is visible in the UI on `/sources/6`.
2. **Calimesa DIF 2017 PDF is image-only.** Solved by adding an OCR
   step (`ocrmypdf`) that produces a text-layer copy
   (`fixtures/calimesa_dif_schedule_2017_ocr.pdf`). All extracted
   numbers from this source are flagged `needs_review`.
3. **Riverside County Fire Marshal and `planning.rctlma.org`** return
   HTTP 403 to plain `requests`, likely WAF / bot-protection. Not
   needed for the demo because the same Fire Protection and
   Transportation amounts are present in the DIF ordinance
   (`rivco_dif_ord_659_14`), which is freely fetchable from
   `rivcocob.org`. Production would add a Playwright fetch path.
4. **Calimesa Master Fee Schedule has tiered "Plan Check" tables.**
   Same fee name (e.g. "Each Additional 100 sq. ft. or fraction there
   of") appears multiple times under different occupancy/construction
   columns. The demo correctly flags these as `conflicting` and stores
   all of them. About 260 of the 2,217 records are in this state;
   they're real fees with legitimate ambiguity that a human still
   needs to disambiguate by reading the column headers.
5. **One source row contains a typo in the public PDF.** The Riverside
   County EH PDF has `"$5, 120.00"` (note the space) for the 36–40
   vendor temporary-event fee. The parser preserves the literal text,
   parses to `5120.00`, lowers confidence, and records a note.
6. **Office row in DIF ordinance drops the dollar sign.** The "Office3"
   rows in the Riverside County DIF tables omit `$` and use bare
   decimals (`0.38 N/A 1.47 …`). The parser handles both forms via a
   tolerant tokenizer; values are still flagged `verified`.

## Sample queries

```sql
-- a few verified DIF Fire Protection fees, by area plan
SELECT fee_name, amount_text, unit, raw_snippet
FROM fees
WHERE category = 'Fire Department'
  AND fee_name LIKE 'DIF — %Fire Protection%'
  AND validation_flag = 'verified'
LIMIT 10;

-- everything OCR or otherwise needing review
SELECT fee_name, amount_text, validation_flag, notes
FROM fees
WHERE validation_flag = 'needs_review'
LIMIT 20;

-- water rates effective 1/1/2026
SELECT fee_name, amount_text, raw_snippet
FROM fees
WHERE category = 'Water';
```

## What's deliberately out of scope

Auth, dashboards, admin tools, multi-tenant separation, change-detection
runs, Postgres infra, scheduled re-extractions, fee-history analytics,
OCR confidence scoring beyond the binary needs_review flag, automated
WAF bypass for the rcfire.org / rctlma.org domains, full mining of the
EMWD PDF beyond the three-column rate rows. All easy to add on top
once the trace-back guarantee is in place — which is what this proof
of concept is for.
